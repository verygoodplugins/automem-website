These endpoints are essential for deployment monitoring, debugging, and understanding the characteristics of stored memories.\n\nFor storing and retrieving memories, see [Memory Operations](/verygoodplugins/automem/5.1-memory-operations). For administrative operations like reprocessing, see [Admin Operations](/verygoodplugins/automem/5.4-consolidation-operations).\n\n* * *\n\n## Overview\n\nAutoMem provides three endpoints for health monitoring and analytics:\n\nEndpoint\n\nAuthentication\n\nPurpose\n\n`GET /health`\n\nNone\n\nService health check with database connectivity and queue status\n\n`GET /analyze`\n\nAPI Token\n\nComprehensive memory graph statistics and patterns\n\n`GET /startup-recall`\n\nNone\n\nRetrieve high-importance memories for initialization context\n\n**Sources:** [app.py2790-3069](https://github.com/verygoodplugins/automem/blob/c2a4dae3/app.py#L2790-L3069)\n\n* * *\n\n## Health Monitoring\n\n### Endpoint: `GET /health`\n\nThe health endpoint provides real-time service status, database connectivity checks, and enrichment pipeline metrics. This endpoint does **not** require authentication and is designed for automated health monitoring systems.\n\n**URL:** `GET /health`\n\n**Authentication:** None required\n\n**Response:** Always returns JSON, HTTP 200 if healthy, HTTP 503 if degraded\n\n#### Response Structure\n\n#### Field Definitions\n\nField\n\nType\n\nDescription\n\n`status`\n\nstring\n\nOverall health: `\"healthy\"` or `\"degraded\"`\n\n`falkordb`\n\nstring\n\nFalkorDB status: `\"connected\"`, `\"unknown\"`, or `\"error: ...\"`\n\n`qdrant`\n\nstring\n\nQdrant status: `\"connected\"`, `\"not_configured\"`, or `\"error: ...\"`\n\n`memory_count`\n\ninteger|null\n\nTotal memories in FalkorDB (null if query fails)\n\n`qdrant_count`\n\ninteger|null\n\nTotal points in Qdrant collection (null if unavailable)\n\n`enrichment`\n\nobject\n\nEnrichment queue metrics (see below)\n\n`graph`\n\nstring\n\nFalkorDB graph name (`FALKORDB_GRAPH` env variable)\n\n`timestamp`\n\nstring\n\nISO 8601 timestamp of health check\n\n#### Enrichment Queue Metrics\n\nThe `enrichment` object provides visibility into the background enrichment pipeline:\n\nField\n\nType\n\nDescription\n\n`status`\n\nstring\n\nWorker state: `\"running\"`, `\"idle\"`, or `\"stopped\"`\n\n`queue_depth`\n\ninteger\n\nTotal jobs in queue (pending + inflight)\n\n`pending`\n\ninteger\n\nJobs waiting to be processed\n\n`inflight`\n\ninteger\n\nJobs currently being processed\n\n`processed`\n\ninteger\n\nTotal jobs completed since service start\n\n`failed`\n\ninteger\n\nTotal jobs that failed permanently\n\n`last_success`\n\nstring|null\n\nTimestamp of most recent successful enrichment\n\n`last_error`\n\nstring|null\n\nMost recent error message (if any)\n\n`worker_active`\n\nboolean\n\nWhether enrichment worker thread is alive\n\n#### Health Check Flow Diagram\n\n#### Example Requests\n\n#### Graceful Degradation\n\nAutoMem continues operating even when components are unavailable:\n\n*   **Qdrant unavailable**: `status` remains `\"healthy\"`, `qdrant` shows `\"not_configured\"` or error\n*   **FalkorDB unavailable**: `status` becomes `\"degraded\"`, HTTP 503 returned\n*   **Enrichment worker stopped**: Service remains healthy but enrichment pipeline stops processing\n\n**Sources:** /analyze`\n\nThe analyze endpoint provides comprehensive statistics about the memory graph, including type distributions, entity frequencies, temporal patterns, and relationship counts. This endpoint is useful for understanding memory characteristics and identifying patterns.\n\n**URL:** `GET /analyze`\n\n**Authentication:** Required (API token via Bearer, `X-API-Key`, or `api_key` query parameter)\n\n**Response:** HTTP 200 with JSON analytics, or HTTP 401 if unauthorized\n\n#### Response Structure\n\n#### Analytics Components\n\nThe `/analyze` endpoint executes 7 independent Cypher queries against FalkorDB to gather different facets of memory data:\n\n1.  **Total Memory Count**: `MATCH (m:Memory) RETURN count(m)`\n2.  **Type Distribution**: Groups memories by `m.type` field\n3.  **Entity Frequency**: Unwinds `m.entities` array and counts occurrences (top 20)\n4.  **Confidence Distribution**: Buckets `m.confidence` scores by 0.1 intervals\n5.  **Activity by Hour**: Extracts hour from `m.timestamp` and counts memories\n6.  **Tag Frequency**: Unwinds `m.tags` array and counts occurrences (top 20)\n7.  **Relationship Counts**: Counts all edges by relationship type\n\nEach query is wrapped in a try-except block—if a query fails, the corresponding field is set to `null`, `{}`, or `[]` depending on the expected type.\n\n#### Analytics Query Flow Diagram\n\n#### Example Requests\n\n#### Use Cases\n\nUse Case\n\nRelevant Fields\n\nIdentify memory class imbalance\n\n`memories_by_type`\n\nFind frequently discussed projects/tools\n\n`top_entities`\n\nAssess memory quality\n\n`confidence_distribution`\n\nUnderstand activity patterns\n\n`activity_by_hour`\n\nAudit tagging consistency\n\n`top_tags`\n\nVerify enrichment pipeline results\n\n`relationships[\"SIMILAR_TO\"]`, `relationships[\"EXEMPLIFIES\"]`\n\nDetect temporal validity issues\n\n`relationships[\"INVALIDATED_BY\"]`, `relationships[\"EVOLVED_INTO\"]`\n\n**Sources:** [app.py2854-3018](https://github.com/verygoodplugins/automem/blob/c2a4dae3/app.py#L2854-L3018) /startup-recall`\n\nThe startup recall endpoint returns a curated set of memories suitable for initializing AI agent context. It prioritizes high-importance memories and falls back to recent memories to ensure the agent has relevant context at startup.\n\n**URL:** `GET /startup-recall`\n\n**Authentication:** None required\n\n**Query Parameters:** None\n\n**Response:** HTTP 200 with JSON memory list, or HTTP 503 if FalkorDB unavailable\n\n#### Response Structure\n\n#### Retrieval Strategy\n\nThe startup recall endpoint uses a two-phase retrieval strategy:\n\n1.  **Phase 1: Trending Memories** (primary)\n    \n    *   Queries: `MATCH (m:Memory) ORDER BY m.importance DESC, m.timestamp DESC LIMIT 10`\n    *   Returns high-importance memories regardless of recency\n    *   Uses `_graph_trending_results()` function\n2.  **Phase 2: Recent Memories** (fallback)\n    \n    *   Only triggered if Phase 1 returns fewer than 10 memories\n    *   Queries: `MATCH (m:Memory) ORDER BY m.timestamp DESC LIMIT (10 - phase1_count)`\n    *   Fills remaining slots with most recent memories\n\n#### Startup Recall Flow Diagram\n\n#### Example Requests\n\n#### Integration with AI Agents\n\nThe startup recall endpoint is designed for AI agent initialization:\n\n**Sources:** [app.py3026-3069](https://github.com/verygoodplugins/automem/blob/c2a4dae3/app.py#L3026-L3069) [app.py669-718](https://github.com/verygoodplugins/automem/blob/c2a4dae3/app.py#L669-L718) (trending function)\n\n* * *\n\n## Monitoring Integration\n\n### Health Monitoring Service\n\nFor production deployments, AutoMem includes a health monitoring service that can detect database drift and trigger automatic recovery:\n\nThe health monitor:\n\n*   Polls `/health` endpoint every 5 minutes (configurable)\n*   Compares `memory_count` vs `qdrant_count` for drift detection\n*   Triggers alerts if drift exceeds 5%\n*   Optionally triggers auto-recovery via `recover_from_qdrant.py`\n\n### Structured Logging\n\nAll three endpoints emit structured logs for observability:\n\n**Sources:** [docs/RAILWAY\\_DEPLOYMENT.md244-273](https://github.com/verygoodplugins/automem/blob/c2a4dae3/docs/RAILWAY_DEPLOYMENT.md#L244-L273) [test-optimizations.sh15-23](https://github.com/verygoodplugins/automem/blob/c2a4dae3/test-optimizations.sh#L15-L23)\n\n* * *\n\n## System Health Dashboard Example\n\n**Sources:** [app.py2790-2852](https://github.com/verygoodplugins/automem/blob/c2a4dae3/app.py#L2790-L2852) [docs/RAILWAY\\_DEPLOYMENT.md256-273](https://github.com/verygoodplugins/automem/blob/c2a4dae3/docs/RAILWAY_DEPLOYMENT.md#L256-L273)\n\nDismiss\n\nRefresh this wiki\n\nThis wiki was recently refreshed. Bearer Token (Recommended)\n\nStandard OAuth 2.0-style authentication using the `Authorization` header:\n\n**Advantages:**\n\n*   Industry standard\n*   Automatically handled by most HTTP libraries\n*   Cannot be logged in server access logs (more secure than query params)\n\n### 2\\. Custom Header\n\nAlternative header-based authentication using `X-API-Key`:\n\n**Use case:** Client libraries that don't support Bearer tokens easily.\n\n### 3\\. Query Parameter\n\nFallback authentication via URL query string:\n\n**⚠️ Security Warning:** Query parameters may be logged by proxies, load balancers, and web servers. Use only when headers are not feasible (e.g., browser-based testing, webhooks with limited configuration).\n\n**Sources:** [INSTALLATION.md337-343](https://github.com/verygoodplugins/automem/blob/c2a4dae3/INSTALLATION.md#L337-L343) [INSTALLATION.md389-396](https://github.com/verygoodplugins/automem/blob/c2a4dae3/INSTALLATION.md#L389-L396)\n\n* * *\n\n## Admin Authentication\n\nAdmin endpoints require **both** the API token and an additional admin token:\n\n**Two-Factor Authentication for Admin Operations**\n\n### Admin-Protected Endpoints\n\nEndpoint\n\nMethod\n\nPurpose\n\n`/admin/reembed`\n\nPOST\n\nRegenerate embeddings in batches\n\n`/enrichment/reprocess`\n\nPOST\n\nRe-queue memories for enrichment\n\n`/enrichment/status`\n\nGET\n\nView enrichment pipeline status (may require admin based on configuration)\n\n**Sources:** [INSTALLATION.md394-396](https://github.com/verygoodplugins/automem/blob/c2a4dae3/INSTALLATION.md#L394-L396) [INSTALLATION.md669-707](https://github.com/verygoodplugins/automem/blob/c2a4dae3/INSTALLATION.md#L669-L707)\n\n* * *\n\n## Configuration\n\n### Environment Variables\n\nConfigure authentication tokens via environment variables loaded in this order:\n\n1. This prevents the Flask API from attempting to connect to FalkorDB before it's ready.\n\n**Dependency Chain**\n\n**Health Check Details**\n\nFalkorDB is the only service with a health check defined [docker-compose.yml16-20](https://github.com/verygoodplugins/automem/blob/c2a4dae3/docker-compose.yml#L16-L20):\n\n*   **Test command**: `redis-cli ping` (expects `PONG` response)\n*   **Interval**: Check every 10 seconds\n*   **Timeout**: Fail if command doesn't respond within 5 seconds\n*   **Retries**: Attempt 5 times before marking unhealthy\n*   **Initial delay**: No explicit start\\_period, begins checking immediately\n\nThe Flask API waits for `condition: service_healthy` [docker-compose.yml51-52](https://github.com/verygoodplugins/automem/blob/c2a4dae3/docker-compose.yml#L51-L52) ensuring FalkorDB is accepting connections before the API attempts to connect.\n\nQdrant uses `condition: service_started` [docker-compose.yml53-54](https://github.com/verygoodplugins/automem/blob/c2a4dae3/docker-compose.yml#L53-L54) meaning the Flask API starts as soon as the Qdrant container starts (not waiting for full initialization).
