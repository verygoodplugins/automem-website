For operational guides, see [Operations](/verygoodplugins/automem/8-operations).\n\n## What is AutoMem?\n\nAutoMem is a Flask-based HTTP API service that provides persistent, queryable memory storage for AI assistants. Unlike traditional RAG systems that retrieve similar documents, AutoMem builds knowledge graphs with typed relationships, enabling multi-hop reasoning, temporal awareness, and pattern learning. The system achieves **90.53% accuracy** on the LoCoMo benchmark (ACL 2024), outperforming CORE (88.24%).\n\n**Key capabilities:**\n\n*   **Store** memories with metadata, importance scores, classification, and semantic embeddings\n*   **Recall** via hybrid search combining vector similarity, keyword matching, graph relationships, and temporal signals\n*   **Connect** memories through 11 typed relationship edges (e.g., `LEADS_TO`, `CONTRADICTS`, `EXEMPLIFIES`)\n*   **Learn** through automatic entity extraction, pattern detection, and neuroscience-inspired consolidation cycles\n*   **Degrade gracefully** when vector search is unavailable, continuing operations in graph-only mode\n\nSources: [README.md1-32](https://github.com/verygoodplugins/automem/blob/c2a4dae3/README.md#L1-L32) [app.py1-7](https://github.com/verygoodplugins/automem/blob/c2a4dae3/app.py#L1-L7) [INSTALLATION.md1-18](https://github.com/verygoodplugins/automem/blob/c2a4dae3/INSTALLATION.md#L1-L18)\n\n* * *\n\n## System Architecture Overview\n\nAutoMem consists of three primary layers: the **MCP Bridge** for cloud AI platform integration, the **Flask API Application Layer** with background workers, and the **Dual-Storage Layer** combining graph and vector databases.\n\n### High-Level Component Diagram\n\n**Note:** Dashed lines to `Qdrant` indicate optional vector search capabilities. implements a dual-storage pattern where **FalkorDB** serves as the canonical record for all memory operations, while **Qdrant** provides optional semantic search capabilities. This architecture enables graceful degradation and built-in redundancy.\n\n### Storage Layer Architecture\n\n**Key principles:**\n\n*   **FalkorDB writes always succeed**: If Qdrant is unavailable, the memory is still persisted in the graph\n*   **Qdrant operations are best-effort**: Failures are logged but don't block API responses\n*   **Full redundancy**: Qdrant payloads contain complete memory properties, enabling recovery from either store\n*   **Automatic sync repair**: `SyncWorker` monitors drift and queues missing embeddings for regeneration\n\nSources: [app.py29-103](https://github.com/verygoodplugins/automem/blob/c2a4dae3/app.py#L29-L103) [INSTALLATION.md349-430](https://github.com/verygoodplugins/automem/blob/c2a4dae3/INSTALLATION.md#L349-L430) [CLAUDE.md70-78](https://github.com/verygoodplugins/automem/blob/c2a4dae3/CLAUDE.md#L70-L78)\n\n* * *\n\n## Request Flow and Memory Lifecycle\n\nThe following diagram illustrates how a memory flows through the system from storage to recall, showing the asynchronous enrichment and consolidation processes.\n\n### Memory Lifecycle Sequence\n\nSources: [app.py193-240](https://github.com/verygoodplugins/automem/blob/c2a4dae3/app.py#L193-L240) [app.py467-900](https://github.com/verygoodplugins/automem/blob/c2a4dae3/app.py#L467-L900) [CLAUDE.md89-96](https://github.com/verygoodplugins/automem/blob/c2a4dae3/CLAUDE.md#L89-L96)\n\n* * *\n\n## Graceful Degradation Design\n\nAutoMem is designed to continue operating even when components fail. Each worker operates on its own queue and tracking sets to prevent duplicate processing.\n\n### Worker Coordination Diagram\n\n**Worker responsibilities:**\n\n*   **EnrichmentWorker**: Extracts entities (spaCy NER + regex), generates summaries, creates temporal/semantic links, detects patterns\n*   **EmbeddingWorker**: Batches requests (20 items, 2-second timeout) to reduce API costs by 40-50%, retries on failure (max 3 attempts)\n*   **ConsolidationScheduler**: Runs four cycles on configurable intervals (Decay, Creative, Cluster, Forget) using control node for persistence\n*   **SyncWorker**: Monitors drift between FalkorDB and Qdrant, auto-repairs when divergence exceeds 5%\n\nSources: [app.py1051-1800](https://github.com/verygoodplugins/automem/blob/c2a4dae3/app.py#L1051-L1800) [INSTALLATION.md404-430](https://github.com/verygoodplugins/automem/blob/c2a4dae3/INSTALLATION.md#L404-L430) [CLAUDE.md86-96](https://github.com/verygoodplugins/automem/blob/c2a4dae3/CLAUDE.md#L86-L96)\n\n* * *\n\n## Embedding Provider Abstraction\n\nAutoMem uses a pluggable provider pattern for embedding generation, supporting multiple backends with automatic failover. The system selects providers in priority order based on availability.\n\n### Provider Selection Logic\n\n**Provider characteristics:**\n\n*   **Voyage**: High-quality embeddings, supports 1024d and 2048d, requires API key\n*   **OpenAI**: High-quality embeddings, supports 768d and 3072d, configurable via `OPENAI_BASE_URL` for compatible providers\n*   **FastEmbed**: Local ONNX models, no API key required, ~210MB model download on first use\n*   **Placeholder**: Hash-based deterministic vectors, no semantic meaning, always available as fallback\n\nSources:
