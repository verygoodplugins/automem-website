Transitions**\n\nA memory moves through these states:\n\n1.  **Created**: `store_memory` returns `memory_id` (stored in both Qdrant vectors and FalkorDB graph)\n2.  **Recalled**: `recall_memory` retrieves memories by semantic/keyword/tag search\n3.  **Associated**: `associate_memories` creates graph edges between memories\n4.  **Updated**: `update_memory` modifies fields (content, tags, importance, metadata)\n5.  **Deleted**: `delete_memory` removes from both databases\n\n**Monitoring** via `check_database_health` is stateless and does not affect memory state.\n\nSources: [src/index.ts882-1031](https://github.com/verygoodplugins/mcp-automem/blob/7e26148d/src/index.ts#L882-L1031) Associate)**\n\n1.  Store a new decision with `store_memory`\n2.  Recall related past decisions with `recall_memory`\n3.  Link to related memories with `associate_memories` (type: `EVOLVED_INTO`, `CONTRADICTS`, `PREFERS_OVER`)\n\n**Pattern 2: Context Recall (Recall at Session Start)**\n\n1.  Session starts → `recall_memory` with project tags and time filter\n2.  Use recalled context to inform responses\n3.  Store new insights with `store_memory` during session\n\n**Pattern 3: Knowledge Evolution (Recall → Update → Associate)**\n\n1.  Recall existing pattern with `recall_memory`\n2.  Update with new insights via `update_memory`\n3.  Create `EVOLVED_INTO` association to track progression\n\nSources: [templates/cursor/automem.mdc.template22-107](https://github.com/verygoodplugins/mcp-automem/blob/7e26148d/templates/cursor/automem.mdc.template#L22-L107) [templates/CLAUDE\\_DESKTOP\\_INSTRUCTIONS.md10-82](https://github.com/verygoodplugins/mcp-automem/blob/7e26148d/templates/CLAUDE_DESKTOP_INSTRUCTIONS.md#L10-L82)\n\n## Tool Registry and Request Handling\n\n### Tool Registration\n\n**Diagram: Tool Registration and Request Flow**\n\nThe MCP server registers tools via two handlers:\n\n1.  **`ListToolsRequestSchema`** (line 874): Returns the `tools` array when clients query available tools\n2.  **`CallToolRequestSchema`** (line 878): Routes tool invocations to implementations via switch statement\n\nEach tool implementation delegates to the corresponding `AutoMemClient` method, which issues HTTP requests to the AutoMem service.\n\nSources: [src/index.ts314-1031](https://github.com/verygoodplugins/mcp-automem/blob/7e26148d/src/index.ts#L314-L1031)\n\n### Request Handler Implementation\n\nThe `CallToolRequestSchema` handler uses a switch statement to route requests:\n\nEach case block:\n\n1.  Casts `args` to the appropriate type (`StoreMemoryArgs`, `RecallMemoryArgs`, etc.)\n2.  Calls the corresponding `AutoMemClient` method\n3.  Formats the response as MCP-compliant JSON\n4.  Returns structured content with `isError` flag on failures\n\nSources: [src/index.ts878-1031](https://github.com/verygoodplugins/mcp-automem/blob/7e26148d/src/index.ts#L878-L1031)\n\n## Content Size Governance\n\n### Two-Tier Limit System\n\nThe `store_memory` tool implements content size governance with soft and hard limits to prevent embedding quality degradation:\n\n**Diagram: Content Size Governance Flow**\n\n**Constants defined at** [src/index.ts887-888](https://github.com/verygoodplugins/mcp-automem/blob/7e26148d/src/index.ts#L887-L888):\n\n**Behavior:**\n\n*   **Below 500 chars**: No warnings, stored as-is\n*   **500-2000 chars**: Warning issued, backend may auto-summarize (decision made by AutoMem service)\n*   **Above 2000 chars**: Immediate rejection with `isError: true` and structured error response\n\nThis prevents large text blocks from degrading embedding quality. It fetches from both:\n\n1.  Semantic search endpoint (`/recall`) - query + tags\n2.  Tag-only endpoint (`/memory/by-tag`) - tag filtering without semantic scoring\n\nResults are merged and deduplicated server-side. The `dedup_removed` count indicates how many duplicates were eliminated.\n\n**Content Size Check Before Storage:**\n\nThe MCP server validates content size BEFORE calling `client.storeMemory()`, preventing oversized content from reaching the backend.\n\nSources: [src/index.ts883-949](https://github.com/verygoodplugins/mcp-automem/blob/7e26148d/src/index.ts#L883-L949) Guidelines\n\n**Importance ranges** as defined in tool descriptions and templates:\n\n*   **0.9-1.0 (Critical)**: Major decisions, breaking changes, user preferences, corrections to AI outputs\n*   **0.75-0.85 (Important)**: Patterns, significant bugs, project context, workflow preferences\n*   **0.5-0.7 (Moderate)**: Minor decisions, helpful context, tool/library choices\n*   **0.3-0.5 (Low)**: Small fixes, temporary workarounds, notes\n*   **< 0.3**: Let decay naturally (not explicitly stored)\n\n**Rationale:** Importance scoring enables filtering during recall and graph expansion via `expand_min_importance`. High-importance memories surface first, while low-importance memories provide breadth when needed.\n\nSources: [src/index.ts362-368](https://github.com/verygoodplugins/mcp-automem/blob/7e26148d/src/index.ts#L362-L368) [templates/CLAUDE\\_DESKTOP\\_INSTRUCTIONS.md84-110](https://github.com/verygoodplugins/mcp-automem/blob/7e26148d/templates/CLAUDE_DESKTOP_INSTRUCTIONS.md#L84-L110) [templates/cursor/automem.mdc.template141-144](https://github.com/verygoodplugins/mcp-automem/blob/7e26148d/templates/cursor/automem.mdc.template#L141-L144)\n\n### Tagging Conventions\n\n**Standard tag structure** from templates:\n\n1.  **Project identifier** (e.g., `\"my-project\"`, `\"automem\"`) - Scopes memories to project\n2.  **Platform tag** (e.g., `\"cursor\"`, `\"claude-code\"`) - Identifies source\n3.  **Time tag** (e.g., `\"2025-01\"`) - Enables time-based filtering\n4.  **Component/domain tags** (e.g., `\"auth\"`, `\"api\"`, `\"frontend\"`) - Categorizes by area\n\n**Tag matching modes** (defined in `recall_memory` schema):\n\n*   `tag_mode: \"any\"` (default): Matches memories with ANY of the provided tags (OR logic)\n*   `tag_mode: \"all\"`: Requires ALL tags to be present (AND logic)\n*   `tag_match: \"exact\"` (default): Exact string match\n*   `tag_match: \"prefix\"`: Prefix matching for namespace hierarchies (e.g., `\"auth/*\"` matches `\"auth/login\"`, `\"auth/jwt\"`)\n\n**Personal vs. Project Namespaces:**\n\nPer [templates/cursor/automem.mdc.template126-137](https://github.com/verygoodplugins/mcp-automem/blob/7e26148d/templates/cursor/automem.mdc.template#L126-L137):\n\n*   **Project memories**: Use project name as first tag (e.g., `[\"my-project\", \"cursor\", \"2025-01\", \"auth\"]`)\n*   **Personal memories**: Use `\"personal\"` tag instead of project (e.g., `[\"personal\", \"2025-01\", \"preferences\"]`)\n\nThis prevents personal preferences and lifestyle context from being drowned out by high-importance technical memories during project-scoped recall.\n\nSources: [src/index.ts475-492](https://github.com/verygoodplugins/mcp-automem/blob/7e26148d/src/index.ts#L475-L492) [templates/cursor/automem.mdc.template126-137](https://github.com/verygoodplugins/mcp-automem/blob/7e26148d/templates/cursor/automem.mdc.template#L126-L137) [templates/codex/memory-rules.md96-97](https://github.com/verygoodplugins/mcp-automem/blob/7e26148d/templates/codex/memory-rules.md#L96-L97)\n\n### Content Format Best Practices\n\n**Recommended format** (150-300 characters):\n\n    Brief title. persists information to the AutoMem service. It covers tool parameters, content size governance, importance scoring, tagging conventions, memory type classification, and metadata patterns.\n\nFor retrieving stored memories, see [Recalling Memories](/verygoodplugins/mcp-automem/4.2-recalling-memories). For creating relationships between memories, see [Associating Memories](/verygoodplugins/mcp-automem/4.3-associating-memories). For updating or deleting memories, see [Managing Memories](/verygoodplugins/mcp-automem/4.4-managing-memories).\n\n* * *\n\n## Overview\n\nThe `store_memory` tool is the primary mechanism for persisting information to the AutoMem memory service. When invoked by an AI platform through the MCP protocol, it:\n\n1.  Validates content size against governance limits (soft: 500 chars, hard: 2000 chars)\n2.  Delegates to `AutoMemClient.storeMemory()` for HTTP transport\n3.  Writes to both FalkorDB (graph relationships) and Qdrant (vector embeddings) in parallel\n4.  Returns a unique `memory_id` for future reference (associations, updates, deletions)\n\nThe tool is designed to be invoked frequently during AI-assisted work, capturing decisions, patterns, insights, and preferences as they emerge rather than attempting comprehensive session-ending summaries.\n\n**Sources:** [src/index.ts319-402](https://github.com/verygoodplugins/mcp-automem/blob/7e26148d/src/index.ts#L319-L402) [src/index.ts882-957](https://github.com/verygoodplugins/mcp-automem/blob/7e26148d/src/index.ts#L882-L957) [src/automem-client.ts102-117](https://github.com/verygoodplugins/mcp-automem/blob/7e26148d/src/automem-client.ts#L102-L117)\n\n* * *\n\n## Tool Definition and Parameters\n\nThe `store_memory` tool is registered in the MCP server with the following schema:\n\n### Required Parameters\n\nParameter\n\nType\n\nDescription\n\n`content`\n\n`string`\n\nThe memory content to store. Measure content length\n    2. If length > HARD_LIMIT (2000):\n       - Return error with isError: true\n       - Include error message with content_length, hard_limit\n       - Do not proceed to storage\n    3. If length > SOFT_LIMIT (500):\n       - Set sizeWarning flag\n       - Continue to storage (backend may summarize)\n    4. If length <= SOFT_LIMIT:\n       - Proceed normally\n    \n\n### Auto-Summarization\n\nWhen content exceeds the soft limit, the backend AutoMem service may automatically summarize it using an LLM. The response includes:\n\n*   `summarized: true` - Flag indicating summarization occurred\n*   `original_length: number` - Original content length\n*   `summarized_length: number` - Post-summarization length\n\n**Sources:** [src/index.ts886-914](https://github.com/verygoodplugins/mcp-automem/blob/7e26148d/src/index.ts#L886-L914) [src/index.ts325-329](https://github.com/verygoodplugins/mcp-automem/blob/7e26148d/src/index.ts#L325-L329)\n\n* * *\n\n## Importance Scoring\n\nImportance scores (0.0 to 1.0) affect recall ranking and natural decay. the primary mechanism for retrieving stored memories from the AutoMem service. It implements hybrid search combining semantic similarity, keyword matching, tag filtering, time-based queries, and graph expansion. For information about storing memories, see [Storing Memories](/verygoodplugins/mcp-automem/4.1-storing-memories). For relationship-based queries, see [Associating Memories](/verygoodplugins/mcp-automem/4.3-associating-memories).\n\n## Overview\n\nThe `recall_memory` tool provides sophisticated retrieval capabilities through the MCP (Model Context Protocol) interface. It delegates to the AutoMem backend service, which performs parallel searches across both Qdrant (vector database) and FalkorDB (graph database) to return comprehensive, scored results.\n\n**Core capabilities:**\n\n*   **Semantic search** via natural language queries\n*   **Tag filtering** with AND/OR logic and prefix matching\n*   **Time-based queries** using natural language or ISO timestamps\n*   **Graph expansion** following typed relationships between memories\n*   **Entity expansion** for multi-hop reasoning\n*   **Context-aware boosting** for programming language and task-specific recall\n\nSources: [src/index.ts403-600](https://github.com/verygoodplugins/mcp-automem/blob/7e26148d/src/index.ts#L403-L600) AutoMem backend generates embeddings and performs vector similarity search in Qdrant. Results are ranked by cosine similarity combined with importance scores.\n\n**Example queries:**\n\n*   `\"authentication timeout issues\"`\n*   `\"React component patterns\"`\n*   `\"database migration decisions\"`\n\n### Tag-Based Search\n\nTag filtering supports two modes and two matching strategies:\n\n**Tag Mode:**\n\n*   `any` (default): Returns memories with ANY of the specified tags (OR logic)\n*   `all`: Requires ALL specified tags (AND logic)\n\n**Tag Match:**\n\n*   `exact` (default): Exact tag match\n*   `prefix`: Supports namespace prefixes like `auth/*` to match `auth/login`, `auth/jwt`, etc.\n\n**Example:**\n\n### Time-Based Queries\n\nTime filtering supports both natural language and ISO timestamp bounds:\n\n**Natural language (`time_query`):**\n\n*   `\"today\"`\n*   `\"yesterday\"`\n*   `\"last week\"`\n*   `\"last 7 days\"`\n*   `\"last 30 days\"`\n\n**Explicit bounds:**\n\n*   `start`: ISO 8601 timestamp for lower bound\n*   `end`: ISO 8601
