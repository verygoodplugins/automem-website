For configuration details, see [Configuration Reference](/verygoodplugins/automem/2.2-configuration-reference).\n\n## Operational Architecture\n\nAutoMem includes several operational services and workflows that ensure data integrity, durability, and system health in production environments.\n\n### Operational Components Overview\n\nSources: [docs/MONITORING\\_AND\\_BACKUPS.md1-583](https://github.com/verygoodplugins/automem/blob/c2a4dae3/docs/MONITORING_AND_BACKUPS.md#L1-L583) [CHANGELOG.md513-660](https://github.com/verygoodplugins/automem/blob/c2a4dae3/CHANGELOG.md#L513-L660)\n\n## Three-Layer Data Protection\n\nAutoMem implements a three-layer backup strategy to ensure data durability and enable rapid recovery from various failure scenarios.\n\n### Backup Layers and Recovery Times\n\nLayer\n\nRecovery Time\n\nSuccess Rate\n\nCost\n\nCoverage\n\n**Layer 1: Railway Volumes**\n\n5 minutes\n\n~100%\n\nIncluded with Pro\n\nFalkorDB only\n\n**Layer 2: Dual Storage**\n\n10 minutes\n\n99.7%\n\nIncluded\n\nBoth databases\n\n**Layer 3: S3 Backups**\n\n30 minutes\n\n~100%\n\n~$0.30/month\n\nBoth databases\n\nSources: [docs/MONITORING\\_AND\\_BACKUPS.md6-48](https://github.com/verygoodplugins/automem/blob/c2a4dae3/docs/MONITORING_AND_BACKUPS.md#L6-L48) [docs/MONITORING\\_AND\\_BACKUPS.md308-335](https://github.com/verygoodplugins/automem/blob/c2a4dae3/docs/MONITORING_AND_BACKUPS.md#L308-L335)\n\n## Health Monitoring and Drift Detection\n\nAutoMem continuously monitors data consistency between FalkorDB and Qdrant to detect drift, which can indicate write failures, data loss, or synchronization issues.\n\n### Drift Detection Architecture\n\n### Health Monitoring Configuration\n\nThe health monitoring system operates at two levels:\n\n**1\\. Internal Sync Worker** (built into Flask service)\n\n*   Runs continuously within the Flask application\n*   Configured via environment variables:\n    *   `SYNC_CHECK_INTERVAL`: Check frequency in seconds (default: 300)\n    *   `SYNC_DRIFT_THRESHOLD`: Warning threshold percentage (default: 5.0)\n*   Automatically queues missing memories for embedding when drift detected\n*   Accessible via `/health` endpoint without authentication\n\n**2\\. External Health Monitor** (standalone script)\n\n*   Runs as separate Railway service or cron job\n*   Script: [scripts/health\\_monitor.py](https://github.com/verygoodplugins/automem/blob/c2a4dae3/scripts/health_monitor.py)\n*   Configuration via environment variables:\n    *   `HEALTH_MONITOR_DRIFT_THRESHOLD`: Warning level (default: 5)\n    *   `HEALTH_MONITOR_CRITICAL_THRESHOLD`: Critical level (default: 50)\n    *   `HEALTH_MONITOR_WEBHOOK`: Webhook URL for alerts\n*   Can trigger automatic recovery when enabled with `--auto-recover` flag\n*   Requires same database credentials as main service\n\nSources: [docs/MONITORING\\_AND\\_BACKUPS.md50-114](https://github.com/verygoodplugins/automem/blob/c2a4dae3/docs/MONITORING_AND_BACKUPS.md#L50-L114) [CHANGELOG.md514-534](https://github.com/verygoodplugins/automem/blob/c2a4dae3/CHANGELOG.md#L514-L534)\n\n### Health Endpoint Response\n\nThe `/health` endpoint returns system status without requiring authentication:\n\nImplementation: [automem/api/health.py](https://github.com/verygoodplugins/automem/blob/c2a4dae3/automem/api/health.py)\n\nSources: [docs/MONITORING\\_AND\\_BACKUPS.md369-389](https://github.com/verygoodplugins/automem/blob/c2a4dae3/docs/MONITORING_AND_BACKUPS.md#L369-L389) [CHANGELOG.md554-579](https://github.com/verygoodplugins/automem/blob/c2a4dae3/CHANGELOG.md#L554-L579)\n\n## Automated Backup System\n\nAutoMem's automated backup system uses GitHub Actions to periodically export both FalkorDB and Qdrant data to compressed JSON files, with optional S3 upload for off-site storage.\n\n### GitHub Actions Backup Workflow\n\n### Critical: TCP Proxy Configuration\n\nGitHub Actions runners execute outside Railway's private network and cannot access internal hostnames like `falkordb.railway.internal`. The backup workflow includes a pre-flight connectivity check that validates TCP Proxy configuration:\n\n**Pre-flight Check** [.github/workflows/backup.yml27-72](https://github.com/verygoodplugins/automem/blob/c2a4dae3/.github/workflows/backup.yml#L27-L72):\n\n1.  Validates `FALKORDB_HOST` does NOT contain `.railway.internal`\n2.  Tests TCP connectivity with 10-second timeout\n3.  Provides detailed error messages if checks fail\n\n**Required GitHub Secrets:**\n\n*   `FALKORDB_HOST`: Must be TCP Proxy domain (e.g., `monorail.proxy.rlwy.net`)\n*   `FALKORDB_PORT`: Must be TCP Proxy port (e.g., `12345`), NOT `6379`\n*   `FALKORDB_PASSWORD`: From Railway FalkorDB service variables\n*   `QDRANT_URL`: Qdrant Cloud HTTPS endpoint\n*   `QDRANT_API_KEY`: Qdrant Cloud API key\n\n**Enabling TCP Proxy:**\n\n1.  Railway Dashboard → `falkordb` service → Settings → Networking\n2.  Enable \"TCP Proxy\" toggle\n3.  Note `RAILWAY_TCP_PROXY_DOMAIN` and `RAILWAY_TCP_PROXY_PORT` values\n4.  Update GitHub secrets accordingly\n\nSources: [.github/workflows/backup.yml1-94](https://github.com/verygoodplugins/automem/blob/c2a4dae3/.github/workflows/backup.yml#L1-L94) [docs/MONITORING\\_AND\\_BACKUPS.md188-277](https://github.com/verygoodplugins/automem/blob/c2a4dae3/docs/MONITORING_AND_BACKUPS.md#L188-L277)\n\n### Backup Schedule and Retention\n\nThe GitHub Actions workflow runs on a cron schedule:\n\nRetention policy (configurable via `--cleanup --keep N`):\n\n*   **Development**: Keep 7 days\n*   **Production**: Keep 14 days + S3 with lifecycle rules\n\nFree tier: GitHub Actions provides 2000 minutes/month, sufficient for ~480 backup runs (20 backups/day for 24 days).\n\nSources: [.github/workflows/backup.yml3-7](https://github.com/verygoodplugins/automem/blob/c2a4dae3/.github/workflows/backup.yml#L3-L7) [docs/MONITORING\\_AND\\_BACKUPS.md243-246](https://github.com/verygoodplugins/automem/blob/c2a4dae3/docs/MONITORING_AND_BACKUPS.md#L243-L246)\n\n## Disaster Recovery Procedures\n\nRecovery time and data completeness depends on which layer of protection is available.\n\n### Recovery Decision Tree\n\n### Recovery Scripts\n\n**1\\. Recover FalkorDB from Qdrant**\n\nScript: [scripts/recover\\_from\\_qdrant.py](https://github.com/verygoodplugins/automem/blob/c2a4dae3/scripts/recover_from_qdrant.py)\n\nRebuilds the FalkorDB graph from Qdrant vectors and payloads. Achieves 99.7% recovery rate (778/780 memories in documented recovery event).\n\n**Process:**\n\n1.  Queries all vectors and payloads from Qdrant collection\n2.  Extracts memory metadata from payloads (content, type, confidence, tags, etc.)\n3.  Filters `RESERVED_FIELDS` to avoid polluting graph with Qdrant-specific metadata\n4.  Recreates `(m:Memory)` nodes in FalkorDB with `MERGE` (idempotent)\n5.  Preserves relationships if they exist in payloads\n\n**Limitations:**\n\n*   Cannot recover relationships not stored in Qdrant payloads\n*   Cannot recover memories without corresponding Qdrant entries\n*   Metadata stored only in FalkorDB (not synced to Qdrant) will be lost\n\n**2\\. Rebuild Qdrant from FalkorDB**\n\nEndpoint: `POST /admin/reembed` [automem/api/admin.py](https://github.com/verygoodplugins/automem/blob/c2a4dae3/automem/api/admin.py)\n\nRe-generates embeddings for all memories and upserts them to Qdrant. Requires `ADMIN_API_TOKEN`.\n\n**Process:**\n\n1.  Queries all memories from FalkorDB\n2.  Batches memories (default 50, configurable)\n3.  Generates embeddings via configured provider\n4.  Upserts to Qdrant with full payloads\n5.  Returns success/failure counts\n\n**3\\. Restore from S3 Backup**\n\nScript: [scripts/restore\\_from\\_backup.py](https://github.com/verygoodplugins/automem/blob/c2a4dae3/scripts/restore_from_backup.py) (to be created if needed)\n\nRestores Qdrant collection from compressed JSON backup.\n\nSources: [docs/MONITORING\\_AND\\_BACKUPS.md306-365](https://github.com/verygoodplugins/automem/blob/c2a4dae3/docs/MONITORING_AND_BACKUPS.md#L306-L365) [CHANGELOG.md663-760](https://github.com/verygoodplugins/automem/blob/c2a4dae3/CHANGELOG.md#L663-L760)\n\n## Performance Optimizations\n\nAutoMem includes several performance optimizations implemented in v0.6.0 and later versions to reduce costs and improve response times.\n\n### Performance Features Overview\n\n### Performance Impact Summary\n\nOptimization\n\nBefore\n\nAfter\n\nImprovement\n\nAnnual Savings\n\n**Embedding Batching**\n\n1000 API calls/day\n\n50-100 API calls/day\n\n40-50% reduction\n\n$8-15/year\n\n**LRU Caching**\n\n~5 min consolidation\n\n~1 min consolidation\n\n80% faster\n\nCPU cost reduction\n\n**Async Processing**\n\n250-400ms POST latency\n\n100-150ms POST latency\n\n60% faster\n\nBetter UX\n\n**Query Timing**\n\nLimited visibility\n\nFull metrics in logs\n\n∞ better\n\nDebugging time\n\n**Configuration Variables:**\n\n*   `EMBEDDING_BATCH_SIZE`: Number of memories per batch (default: 20)\n*   `EMBEDDING_BATCH_TIMEOUT_SECONDS`: Max wait time to collect batch (default: 2.0)\n*   `ENRICHMENT_IDLE_SLEEP`: Sleep time when queue empty (default: 1.0)\n*   `SYNC_CHECK_INTERVAL`: Sync worker check frequency (default: 300)\n\n**Implementation Details:**\n\n*   Embedding batching: [app.py](https://github.com/verygoodplugins/automem/blob/c2a4dae3/app.py) - `_generate_real_embeddings_batch()` function\n*   LRU cache: [automem/consolidation/engine.py](https://github.com/verygoodplugins/automem/blob/c2a4dae3/automem/consolidation/engine.py) - `_get_relationship_count_cached_impl()`\n*   Async queues: [app.py](https://github.com/verygoodplugins/automem/blob/c2a4dae3/app.py) - `ServiceState.embedding_queue`, `ServiceState.enrichment_queue`\n*   Query timing: [automem/api/](https://github.com/verygoodplugins/automem/blob/c2a4dae3/automem/api/) - All endpoint modules include `query_time_ms` in responses\n\nSources: [CHANGELOG.md513-660](https://github.com/verygoodplugins/automem/blob/c2a4dae3/CHANGELOG.md#L513-L660) [docs/MONITORING\\_AND\\_BACKUPS.md513-534](https://github.com/verygoodplugins/automem/blob/c2a4dae3/docs/MONITORING_AND_BACKUPS.md#L513-L534)\n\n## Common Operational Issues\n\n### Issue: ECONNREFUSED When Services Start\n\n**Symptom:** MCP bridge or other services fail to connect to Flask API with `ECONNREFUSED` error.\n\n**Root Causes:**\n\n1.  `PORT=8001` environment variable not set (Flask defaults to 5000)\n2.  Flask binding to `0.0.0.0` instead of `::` (IPv6 dual-stack required for Railway)\n3.  Service name mismatch in `AUTOMEM_ENDPOINT`\n\n**Solutions:**\n\n1.  Verify `PORT=8001` is set on memory-service in Railway Dashboard\n2.  Verify Flask binds to `::` (dual-stack): [app.py](https://github.com/verygoodplugins/automem/blob/c2a4dae3/app.py) - `app.run(host='::', port=port)`\n3.  Verify `AUTOMEM_ENDPOINT` uses correct internal DNS: `http://memory-service.railway.internal:8001`\n\n**Validation:**\n\nSources: [CHANGELOG.md340-431](https://github.com/verygoodplugins/automem/blob/c2a4dae3/CHANGELOG.md#L340-L431) [docs/RAILWAY\\_DEPLOYMENT.md](https://github.com/verygoodplugins/automem/blob/c2a4dae3/docs/RAILWAY_DEPLOYMENT.md) (referenced in changelog)\n\n### Issue: GitHub Actions Backup Connection Failed\n\n**Symptom:** Backup workflow fails with \"Connection reset by peer\" or timeout.\n\n**Root Cause:** GitHub Actions cannot access Railway internal hostnames (`.railway.internal`). Must use TCP Proxy for external connectivity.\n\n**Solution:**\n\n1.  Enable TCP Proxy on FalkorDB service (Railway Dashboard → Networking)\n2.  Update GitHub secrets:\n    *   `FALKORDB_HOST`: Use TCP Proxy domain (e.g., `monorail.proxy.rlwy.net`)\n    *   `FALKORDB_PORT`: Use TCP Proxy port (e.g., `12345`), NOT `6379`\n3.  Run workflow again\n\n**Pre-flight checks in workflow:** [.github/workflows/backup.yml27-72](https://github.com/verygoodplugins/automem/blob/c2a4dae3/.github/workflows/backup.yml#L27-L72)\n\nSources: [docs/MONITORING\\_AND\\_BACKUPS.md248-277](https://github.com/verygoodplugins/automem/blob/c2a4dae3/docs/MONITORING_AND_BACKUPS.md#L248-L277) [.github/workflows/backup.yml27-72](https://github.com/verygoodplugins/automem/blob/c2a4dae3/.github/workflows/backup.yml#L27-L72)\n\n### Issue: High Drift Between FalkorDB and Qdrant\n\n**Symptom:** `/health` endpoint shows `drift_percent > 5%`.\n\n**Root Causes:**\n\n1.  Normal in-flight writes (<1% drift) - no action needed\n2.  Failed Qdrant writes during high load (5-10% drift) - warning\n3.  Qdrant service interruption or connectivity issue (>10% drift) - critical\n\n**Automated Response:**\n\n*   Sync worker automatically queues missing memories for embedding when drift detected\n*   Implementation: [automem/workers/sync.py](https://github.com/verygoodplugins/automem/blob/c2a4dae3/automem/workers/sync.py) - detects drift, adds to `embedding_queue`\n\n**Manual Recovery (if automated sync fails):**\n\nSources: [docs/MONITORING\\_AND\\_BACKUPS.md516-535](https://github.com/verygoodplugins/automem/blob/c2a4dae3/docs/MONITORING_AND_BACKUPS.md#L516-L535)\n\n### Issue: Enrichment Queue Backlog\n\n**Symptom:** `/health` shows `enrichment.queue_depth` growing continuously.\n\n**Root Causes:**\n\n1.  Entity extraction taking too long (spaCy NLP processing)\n2.  Enrichment worker crashed or not running\n3.  High memory creation rate overwhelming worker\n\n**Diagnosis:**\n\n**Solutions:**\n\n1.  Increase `ENRICHMENT_IDLE_SLEEP` to reduce CPU usage during idle (trade: slower processing)\n2.  Restart service to recover crashed worker\n3.  Manually reprocess failed memories: `POST /enrichment/reprocess` with `ADMIN_API_TOKEN`\n\n**Implementation:** [automem/workers/enrichment.py](https://github.com/verygoodplugins/automem/blob/c2a4dae3/automem/workers/enrichment.py)\n\nSources: [CHANGELOG.md554-579](https://github.com/verygoodplugins/automem/blob/c2a4dae3/CHANGELOG.md#L554-L579)\n\n## Operational Metrics\n\nAutoMem exposes several metrics for monitoring production Volume Snapshots\n\nRailway provides automatic volume backups for the FalkorDB persistent volume configured in the deployment template.\n\n**Volume Configuration:**\n\nThe FalkorDB service uses a persistent volume mounted at `/var/lib/falkordb/data` as specified in [railway-template.json54-58](https://github.com/verygoodplugins/automem/blob/c2a4dae3/railway-template.json#L54-L58):\n\n**Persistence Configuration:**\n\nThe Redis persistence settings in [railway-template.json70](https://github.com/verygoodplugins/automem/blob/c2a4dae3/railway-template.json#L70-L70) ensure data durability:\n\nThis configures:\n\n*   RDB snapshots every 60 seconds if at least 1 write\n*   AOF (Append-Only File) for write-ahead logging\n*   `fsync` every second for durability\n\n**Access and Restore:**\n\n1.  Railway Dashboard → FalkorDB service\n2.  \"Backups\" tab shows snapshot history\n3.  One-click restore from any snapshot\n\n**Limitations:**\n\n*   Only covers FalkorDB (Qdrant not included)\n*   Cannot export/download backups\n*   Platform-locked to Railway\n*   Best for quick recovery from recent failures\n\n**Sources:** [docs/MONITORING\\_AND\\_BACKUPS.md84-104](https://github.com/verygoodplugins/automem/blob/c2a4dae3/docs/MONITORING_AND_BACKUPS.md#L84-L104) Structure:**\n\nThe FalkorDB export captures the entire Redis keyspace including:\n\n*   Memory nodes with all properties\n*   Relationship edges\n*   Metadata and indices\n*   Graph structure information\n\n**Qdrant Export Structure:**\n\nThe Qdrant export includes:\n\n*   Vector embeddings (768-dimensional)\n*   Payload data (memory content, metadata, tags)\n*   Point IDs mapped to memory IDs\n*   Collection configuration\n\nBoth exports are compressed using `gzip` with `.json.gz` extension, typically achieving 70-80% compression ratio.\n\n**Sources:** metadata\n\n**Relationships:**\n\n*   Relationship type (RELATES\\_TO, PREFERS\\_OVER, etc.)\n*   Source and target node IDs\n*   Relationship properties (strength, created\\_at)\n\n**Graph Metadata:**\n\n*   Index definitions\n*   Constraint definitions\n*   Graph statistics\n\n**Example Structure:**\n\n**Sources:** captures all vector embeddings and their associated metadata by scrolling through the collection.\n\n**Point Data:**\n\n*   Point ID (matches memory ID)\n*   Vector (768-dimensional float array)\n*   Payload data:\n    *   `content`: Original memory text\n    *   `memory_id`: UUID from FalkorDB\n    *   `tags`: Array of tag strings\n    *   `importance`: Float 0.0-1.0\n    *   `type`: Memory classification\n    *   `created_at`: Timestamp\n    *   Additional metadata fields\n\n**Collection Metadata:**\n\n*   Collection name\n*   Vector size (768)\n*   Distance metric (Cosine)\n*   Point count\n\n**Example Structure:**\n\n**Sources:** [docs/MONITORING\\_AND\\_BACKUPS.md109-124](https://github.com/verygoodplugins/automem/blob/c2a4dae3/docs/MONITORING_AND_BACKUPS.md#L109-L124)\n\n* *
