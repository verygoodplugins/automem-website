Routes are organized by functional area.\n\n### Endpoint Categories\n\nCategory\n\nRoute Pattern\n\nHandler Function\n\nAuth Required\n\n**Health**\n\n`GET /health`\n\n`health_check()`\n\nNo\n\n**Memory CRUD**\n\n`POST /memory`\n\n`post_memory()`\n\nYes\n\n`GET /memory/<id>`\n\n`get_memory()`\n\nYes\n\n`PATCH /memory/<id>`\n\n`patch_memory()`\n\nYes\n\n`DELETE /memory/<id>`\n\n`delete_memory()`\n\nYes\n\n`GET /memory/by-tag`\n\n`get_memory_by_tag()`\n\nYes\n\n**Recall**\n\n`GET /recall`\n\n`recall_memories()`\n\nYes\n\n`GET /startup-recall`\n\n`startup_recall()`\n\nYes\n\n`GET /memories/<id>/related`\n\n`get_related_memories()`\n\nYes\n\n**Association**\n\n`POST /associate`\n\n`create_association()`\n\nYes\n\n**Enrichment**\n\n`GET /enrichment/status`\n\n`enrichment_status()`\n\nYes\n\n`POST /enrichment/reprocess`\n\n`reprocess_enrichment()`\n\nAdmin\n\n**Admin**\n\n`POST /admin/reembed`\n\n`admin_reembed()`\n\nAdmin\n\n**Consolidation**\n\n`POST /consolidate`\n\n`trigger_consolidation()`\n\nYes\n\n`GET /consolidate/status`\n\n`consolidation_status()`\n\nYes\n\n**Analytics**\n\n`GET /analyze`\n\n`analyze_memories()`\n\nYes\n\n**Sources:** [docs/API.md15-119](https://github.com/verygoodplugins/automem/blob/c2a4dae3/docs/API.md#L15-L119) [app.py1500-2200](https://github.com/verygoodplugins/automem/blob/c2a4dae3/app.py#L1500-L2200) (route handler implementations)\n\n### Route Handler Pattern\n\nAll route handlers follow a consistent pattern:\n\n**Sources:** [app.py1500-2200](https://github.com/verygoodplugins/automem/blob/c2a4dae3/app.py#L1500-L2200) (examples: post\\_memory, recall\\_memories, patch\\_memory)\n\n* * *\n\n## Store Integration\n\nThe Flask API coordinates between FalkorDB (graph) and Qdrant (vector) with a dual-write pattern and graceful degradation.\n\n### Dual-Write Pattern\n\n**Critical Design:** Graph writes always succeed or fail the request. Vector writes are best-effort—failures are logged but don't block responses.\n\n**Sources:** [app.py1501-1750](https://github.com/verygoodplugins/automem/blob/c2a4dae3/app.py#L1501-L1750) (post\\_memory implementation)\n\n### Graceful Degradation\n\nWhen Qdrant is unavailable (`state.qdrant is None`), the system operates in graph-only mode:\n\nOperation\n\nGraph-Only Behavior\n\n**Storage**\n\nStores in FalkorDB, skips vector embedding\n\n**Recall**\n\nFalls back to keyword + tag search in graph\n\n**Update**\n\nUpdates graph only, logs Qdrant skip\n\n**Delete**\n\nDeletes from graph only\n\n**Health**\n\nReturns `status: \"degraded\"`\n\n**Sources:** [app.py341-349](https://github.com/verygoodplugins/automem/blob/c2a4dae3/app.py#L341-L349) (health endpoint), [app.py612-683](https://github.com/verygoodplugins/automem/blob/c2a4dae3/app.py#L612-L683) (vector\\_search fallback)\n\n* * *\n\n## Background Worker Coordination\n\nThe Flask API enqueues asynchronous work for background threads without blocking HTTP responses.\n\n### Queue Management\n\n**Sources:** [app.py1051-1160](https://github.com/verygoodplugins/automem/blob/c2a4dae3/app.py#L1051-L1160) (ServiceState queue definitions), [app.py1501-1750](https://github.com/verygoodplugins/automem/blob/c2a4dae3/app.py#L1501-L1750) (queue.put calls)\n\n### Enqueue Pattern\n\nRoute handlers use a consistent pattern to enqueue background work:\n\n**Deduplication guarantee:** Each memory is processed at most once per worker type. The `pending` and `inflight` sets prevent duplicate queue entries.\n\n**Sources:** [app.py1680-1720](https://github.com/verygoodplugins/automem/blob/c2a4dae3/app.py#L1680-L1720) (enrichment queueing), [app.py1730-1750](https://github.com/verygoodplugins/automem/blob/c2a4dae3/app.py#L1730-L1750) (embedding queueing)\n\n* * *\n\n## Error Handling\n\nThe Flask API implements multiple layers of error handling with structured JSON responses.\n\n### Error Response Format\n\nAll errors return JSON with consistent structure:\n\n### Error Categories\n\n**Sources:** [app.py2850-2900](https://github.com/verygoodplugins/automem/blob/c2a4dae3/app.py#L2850-L2900) (error handlers)\n\n### Graceful Degradation Strategy\n\nThe application detects service availability and adjusts behavior:\n\nCondition\n\nDetection\n\nResponse\n\n**FalkorDB unavailable**\n\n`state.memory_graph is None`\n\nReturn 503 for all memory operations\n\n**Qdrant unavailable**\n\n`state.qdrant is None`\n\nContinue with graph-only mode, log warnings\n\n**OpenAI unavailable**\n\n`state.openai_client is None`\n\nUse placeholder embeddings or skip classification\n\n**spaCy model missing**\n\n`_SPACY_NLP is None`\n\nFall back to regex-based entity extraction\n\n**Critical principle:** FalkorDB is the canonical data store. If graph writes fail, the entire operation fails. Vector writes are best-effort.\n\n**Sources:** [app.py1267-1440](https://github.com/verygoodplugins/automem/blob/c2a4dae3/app.py#L1267-L1440) (init functions with try-except), [app.py341-349](https://github.com/verygoodplugins/automem/blob/c2a4dae3/app.py#L341-L349) (health check degradation)\n\n* * *\n\n## Memory Classifier Integration\n\nThe Flask API uses `MemoryClassifier` to automatically detect memory types when not explicitly provided.\n\n### Classification Flow\n\n**Sources:** [app.py685-851](https://github.com/verygoodplugins/automem/blob/c2a4dae3/app.py#L685-L851) (MemoryClassifier class)\n\n### Pattern Matching\n\nThe classifier first attempts regex-based classification (fast, free):\n\nIf multiple patterns match, confidence increases: `min(0.95, 0.6 + (matches * 0.1))`\n\n**Sources:** [app.py687-752](https://github.com/verygoodplugins/automem/blob/c2a4dae3/app.py#L687-L752) (pattern definitions), [app.py766-801](https://github.com/verygoodplugins/automem/blob/c2a4dae3/app.py#L766-L801) (classification logic)\n\n* * *\n\n## Configuration Management\n\nThe Flask API imports all configuration from `automem.config` to ensure consistency across modules.\n\n### Configuration Imports\n\n**No hardcoded configuration:** All environment-dependent values are imported from `automem.config`, making the system testable and deployment-agnostic.\n\n**Sources:** [app.py122-171](https://github.com/verygoodplugins/automem/blob/c2a4dae3/app.py#L122-L171) (configuration imports), [automem/config.py1-200](https://github.com/verygoodplugins/automem/blob/c2a4dae3/automem/config.py#L1-L200) (configuration definitions)\n\n* * *\n\n## Logging and Observability\n\nThe Flask API implements structured logging to stdout for Railway compatibility.\n\n### Logging Configuration\n\n**Railway-specific requirement:** Logs written to stderr are categorized as errors in Railway's UI. Writing to stdout ensures correct log level parsing.\n\n**Sources:** [app.py86-103](https://github.com/verygoodplugins/automem/blob/c2a4dae3/app.py#L86-L103) (logging configuration)\n\n### SSE Event Streaming\n\nThe API supports Server-Sent Events (SSE) for real-time observability:\n\n**Sources:** [app.py74](https://github.com/verygoodplugins/automem/blob/c2a4dae3/app.py#L74-L74) (SSE import), [automem/api/stream.py1-100](https://github.com/verygoodplugins/automem/blob/c2a4dae3/automem/api/stream.py#L1-L100) (SSE implementation)\n\n* * *\n\n## Summary\n\nThe Flask API Service provides:\n\n1.  **HTTP REST Interface** - 28 core endpoints for memory operations\n2.  **ServiceState Management** - Singleton pattern for shared connections and queues\n3.  **Authentication Middleware** - Two-tier token validation (standard + admin)\n4.  **Dual-Write Coordination** - FalkorDB (canonical) + Qdrant (best-effort)\n5.  **Background Worker Integration** - Non-blocking queue-based async processing\n6.  **Graceful Degradation** - Graph-only mode when Qdrant unavailable\n7.  **Structured Error Handling** - Consistent JSON responses with proper HTTP codes\n8.  **Memory Classification** - Regex + LLM hybrid type detection\n9.  **Observability** - Structured logging and SSE streaming\n\nThe application is deployed via Gunicorn on port 8001 with `::` binding for Railway's IPv6 internal networking.\n\n**Sources:** [app.py1-3500](https://github.com/verygoodplugins/automem/blob/c2a4dae3/app.py#L1-L3500) [README.md333-362](https://github.com/verygoodplugins/automem/blob/c2a4dae3/README.md#L333-L362) (deployment), [docs/API.md1-126](https://github.com/verygoodplugins/automem/blob/c2a4dae3/docs/API.md#L1-L126) (endpoint reference)\n\nDismiss\n\nRefresh this wiki\n\nThis wiki was recently refreshed. [.env.example](https://github.com/verygoodplugins/automem/blob/c2a4dae3/.env.example)\n*   [CLAUDE.md](https://github.com/verygoodplugins/automem/blob/c2a4dae3/CLAUDE.md)\n*   [README.md](https://github.com/verygoodplugins/automem/blob/c2a4dae3/README.md)\n*   [app.py](https://github.com/verygoodplugins/automem/blob/c2a4dae3/app.py)\n\nThis document provides an overview of AutoMem's three asynchronous background processing systems that operate independently of the main Flask API request/response cycle. These systems handle computationally expensive operations without blocking client requests.\n\nFor detailed implementation of each system, see:\n\n*   [Enrichment Pipeline](/verygoodplugins/automem/4.3.1-enrichment-pipeline) - Entity extraction and relationship building\n*   [Embedding Generation](/verygoodplugins/automem/4.3.2-embedding-generation) - Batched background processing systems, each with distinct triggers, execution models, and responsibilities:\n\nSystem\n\nTrigger\n\nExecution Model\n\nPrimary Purpose\n\n**Enrichment
