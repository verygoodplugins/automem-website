The 500-character soft limit signals to the backend that summarization may be beneficial, while the 2000-character hard limit prevents storage of entire code files or documentation.\n\n**Sources**: [src/index.ts205-225](https://github.com/verygoodplugins/mcp-automem/blob/7e26148d/src/index.ts#L205-L225) [CHANGELOG.md56-57](https://github.com/verygoodplugins/mcp-automem/blob/7e26148d/CHANGELOG.md#L56-L57) [templates/cursor/automem.mdc.template256-266](https://github.com/verygoodplugins/mcp-automem/blob/7e26148d/templates/cursor/automem.mdc.template#L256-L266)\n\n## Importance Level Guidelines and Memory Types\n\n### Memory Type Classification\n\nThe system supports seven memory types with distinct semantic meanings:\n\nType\n\nUsage\n\nImportance Range\n\nExample\n\n`Decision`\n\nStrategic/technical decisions\n\n0.85-0.95\n\n\"Chose PostgreSQL over MongoDB for ACID compliance\"\n\n`Pattern`\n\nRecurring approaches\n\n0.70-0.85\n\n\"Using early returns for validation reduces nesting\"\n\n`Insight`\n\nKey learnings, resolutions\n\n0.75-0.85\n\n\"Auth timeout caused by missing retry logic\"\n\n`Preference`\n\nUser/team preferences\n\n0.75-0.90\n\n\"User prefers 2-space indents with single quotes\"\n\n`Style`\n\nCode style, formatting\n\n0.60-0.80\n\n\"Always wrap database calls with timeout logging\"\n\n`Habit`\n\nRegular behaviors\n\n0.50-0.70\n\n\"Deploy to staging before production\"\n\n`Context`\n\nGeneral information\n\n0.50-0.70\n\n\"Added JWT authentication system\"\n\n**Type vs Importance**: The `type` field enables semantic filtering (e.g., recall only `Decision` memories), while `importance` controls relevance ranking and decay. Corrections to AI outputs should always use `importance: 0.9` as they represent strong style signals.\n\n**Sources**: [templates/CLAUDE\\_MD\\_MEMORY\\_RULES.md53-58](https://github.com/verygoodplugins/mcp-automem/blob/7e26148d/templates/CLAUDE_MD_MEMORY_RULES.md#L53-L58) [templates/cursor/automem.mdc.template109-123](https://github.com/verygoodplugins/mcp-automem/blob/7e26148d/templates/cursor/automem.mdc.template#L109-L123) [templates/codex/memory-rules.md33-41](https://github.com/verygoodplugins/mcp-automem/blob/7e26148d/templates/codex/memory-rules.md#L33-L41)\n\n### Importance Scoring Strategy\n\n**Philosophy**: High importance scores (0.9+) are reserved for memories that should persist indefinitely and rank highly in recall. Medium scores (0.7-0.8) indicate useful patterns that should survive for months. Low scores (< 0.3) signal the memory system to let them naturally decay.\n\n**Sources**: [templates/cursor/automem.mdc.template140-144](https://github.com/verygoodplugins/mcp-automem/blob/7e26148d/templates/cursor/automem.mdc.template#L140-L144) [templates/CLAUDE\\_MD\\_MEMORY\\_RULES.md54](https://github.com/verygoodplugins/mcp-automem/blob/7e26148d/templates/CLAUDE_MD_MEMORY_RULES.md#L54-L54) [templates/CLAUDE\\_DESKTOP\\_INSTRUCTIONS.md84-109](https://github.com/verygoodplugins/mcp-automem/blob/7e26148d/templates/CLAUDE_DESKTOP_INSTRUCTIONS.md#L84-L109)\n\n## Tagging Conventions and Namespaces\n\n### Project-Specific vs Personal Tagging\n\n**Rationale**: Project tags (`my-app`, `authentication`, etc.) enable precise filtering for technical work. The `personal` tag creates a separate namespace for cross-project memories like user preferences, ensuring they aren't drowned out by high-importance technical memories during recall.\n\n**Tag matching modes**:\n\n*   `tag_mode: \"any\"` (default) - Matches memories with ANY of the provided tags (OR logic)\n*   `tag_mode: \"all\"` - Requires ALL tags to be present (AND logic)\n*   `tag_match: \"prefix\"` - Supports namespace hierarchies like `auth/*` matching `auth/jwt`, `auth/oauth`\n\n**Sources**: Worked\n\n1.  **Queue-based processing concept**: Deferring API calls to batch operations remains useful (CLI still supports `queue` command)\n2.  **Profile system**: Opt-in complexity was a good should see and understand what's stored\n\n**Stable protocols**\n\nDepend on documented APIs (MCP), not internal hooks\n\n**Simple architecture**\n\nFewer moving parts = fewer bugs\n\n**Inline processing**\n\nReal-time feedback better than async background jobs\n\n### Alternative Approaches Considered\n\n**Agent-based capture**: Let AI assistants with extended context proactively store memories during work. Verify no CLI arguments passed (would trigger CLI mode instead of server mode [src/index.ts37](https://github.com/verygoodplugins/mcp-automem/blob/7e26148d/src/index.ts#L37-L37)).\n\n**Tools not appearing**\n\nPlatform doesn't list memory tools\n\nVerify server name in platform config. Check if platform requires tool allowlist. Restart platform after config changes.\n\n**Authentication failures**\n\n401/403 errors\n\nSet `AUTOMEM_API_KEY` in server environment variables, not client-side. Verify API key format matches backend requirements.\n\n**Recall returns empty**\n\nQueries return 0 results despite stored memories\n\nCheck tags match exactly. Verify time filters aren't too restrictive. Try query without tags first. Check backend health.\n\n**Content too large errors**\n\nStore operations rejected\n\nEnforce 500-char soft limit, 2000-char hard limit [src/index.ts887-888](https://github.com/verygoodplugins/mcp-automem/blob/7e26148d/src/index.ts#L887-L888) before calling API. Split long content into multiple memories with associations.\n\n**Timeout errors**\n\nOperations fail after ~25 seconds\n\nReduce `limit` parameter in recall queries. Check backend performance. Consider using `searchByTag` instead of semantic search for simple tag queries.\n\n**Sources:** [src/index.ts32-318](https://github.com/verygoodplugins/mcp-automem/blob/7e26148d/src/index.ts#L32-L318) [src/automem-client.ts22-100](https://github.com/verygoodplugins/mcp-automem/blob/7e26148d/src/automem-client.ts#L22-L100) [src/index.ts886-914](https://github.com/verygoodplugins/mcp-automem/blob/7e26148d/src/index.ts#L886-L914)\n\n### Debugging Techniques\n\n**1\\.
